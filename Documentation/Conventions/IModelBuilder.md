# IModel Builder Convention

Any class or record implementing `IModel` must have a matching builder named:

`{ModelName}Builder`

For example:

`Country : IModel` requires `CountryBuilder`.

## Required Methods

### Create(...)

Creates a normal application instance.

- Accepts the `UserID` and the domain model properties needed to construct the model.
- Does not accept `AuditDateTime` directly.
- Uses `AuditDateTimeBuilder.Create()` internally.
- Creates the corresponding `{ModelName}CreatedEvent`.
- Uses the created event to build the model by calling `Create({ModelName}CreatedEvent createdEvent)`.

### CreateSeed(...)

Creates seed/setup data.

- Accepts the domain model properties needed to construct the model.
- Does not require `UserID`.
- Uses `Constants.Initialisation.UserID`.
- Uses setup constants where appropriate, such as `Constants.Initialisation.EventDateTime`, `Constants.Initialisation.AuditDateTime`, and `Constants.Initialisation.Reason`.
- Creates the corresponding `{ModelName}CreatedEvent` through `{ModelName}CreatedEventBuilder.CreateSeed(...)`.
- Uses the created event to build the model by calling `Create({ModelName}CreatedEvent createdEvent)`.

### Create({ModelName}CreatedEvent createdEvent)

Creates the model from its created event.

- The created event must implement `IEventBase`.
- The model should take its valuation, audit, and event provenance from the event:
  - `ValuationDateTime = createdEvent.EventDateTime`
  - `AsOfDateTime = createdEvent.AuditDateTime`
  - `LastEventID = createdEvent.EventID`
  - `LastAuditDateTime = createdEvent.AuditDateTime`

## Event-Backed Changes

When a model property is represented by a separate modification event, do not include that property in `Create(...)` or `CreateSeed(...)`.

For example, `Country.Flag` is set by `CountryFlagModifiedEvent`, so `CountryBuilder.Create(...)` and `CountryBuilder.CreateSeed(...)` should create only the country identity/details represented by `CountryCreatedEvent`.

## Constructor And Validation Boundaries

Models implementing `IModel` are products of applying events. Their constructors should assign event-derived state and should not be the primary business validation boundary.

Events are the validation boundary:

- Each event should have a private `[JsonConstructor]` constructor for deserialization.
- Each event should have a public constructor used by the normal builder path.
- The public constructor should validate required metadata and domain payload.
- Normal event builders should not accept `EventID` or `AuditDateTime`; those are generated by the event constructor.
- Seed/setup event builders should use `CreateSeed(...)` when fixed setup metadata is required.
